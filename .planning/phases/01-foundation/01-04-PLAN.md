Create `.github/workflows/ci.yml` with EXACTLY this content (verbatim from RESEARCH.md §"GitHub Actions Workflow"):

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize, reopened, ready_for_review]
  workflow_dispatch:

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build-test-lint:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [20, 22]
    runs-on: ${{ matrix.os }}
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Typecheck
        run: npm run typecheck

      - name: Lint
        run: npm run lint

      - name: Build
        run: npm run build

      - name: Test
        run: npm test

      - name: Smoke check — loci --version
        run: node dist/cli.mjs --version
```

Point-by-point rationale (DO NOT deviate — each choice is locked by CONTEXT.md or RESEARCH.md):

1. **`name: CI`** — short and recognisable in the GitHub Actions UI
2. **`on.push.branches: [main]`** — D-12: trigger on push to main only (feature branches get CI via their PR)
3. **`on.pull_request.types`** — includes `ready_for_review` so draft PRs trigger CI the moment they're marked ready (standard best practice)
4. **`on.workflow_dispatch`** — D-12: manual dispatch for re-runs
5. **`concurrency.group: ci-${{ github.ref }}`** + `cancel-in-progress: true` — avoid stacking runs on force-push; saves CI minutes
6. **`fail-fast: false`** — D-10 cross-platform requirement: if Windows fails but Ubuntu passes, we still want to see the Ubuntu result
7. **`matrix.os: [ubuntu-latest, windows-latest, macos-latest]`** — D-10: `windows-latest` = Windows Server 2022 (real Windows, not WSL); `macos-latest` = Apple Silicon
8. **`matrix.node: [20, 22]`** — D-09: v20 = engines floor, v22 = current Active LTS; Node 24 excluded
9. **`actions/checkout@v4`** — current stable
10. **`actions/setup-node@v4` with `cache: npm`** — built-in npm cache keyed on package-lock.json; no separate actions/cache step
11. **`npm ci`** — reproducible install from lockfile; REQUIRES package-lock.json committed by Plan 01 (RESEARCH.md Pitfall 6)
12. **Step order: typecheck → lint → build → test → smoke** —
    - **Typecheck first** — fails fastest on type regressions
    - **Lint** — catches style before the slower build
    - **Build** — produces dist/cli.mjs that the smoke + E2E tests need
    - **Test** — includes cli.e2e.test.ts which spawns dist/cli.mjs; must run AFTER build
    - **Smoke** — the last gate, directly executes the bundled binary. On Windows this runs via `node dist/cli.mjs --version` (not via the cmd shim) — verifies the shebang-bearing bundle runs in PowerShell/cmd.exe
13. **No `permissions:` block** — default read-only token is sufficient (CI doesn't push, tag, publish, or comment)
14. **NO hyperfine step** — D-11 defers the <300ms cold-start gate to Phase 5. The smoke check is exit-code-only, not a timing assertion. DO NOT add `hyperfine` to this workflow.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub → CI runner | Runner images are public Microsoft-hosted VMs (ubuntu/windows/macos) |
| CI runner → npm registry | `npm ci` fetches deps from registry; pinned by lockfile |
| CI workflow → GITHUB_TOKEN | Default read-only token; no write permissions needed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | Supply chain via `npm ci` in CI | mitigate | Runtime deps pinned EXACTLY in package.json (no caret); package-lock.json committed; `npm ci` fails on lockfile drift (Pitfall 6). GitHub runners use Microsoft-hosted VMs with fresh state per run — no persistent `npm cache` contamination. |
| T-04-02 | Elevation of Privilege | Malicious code in third-party GitHub Action | mitigate | Only two actions used: `actions/checkout@v4` and `actions/setup-node@v4` — both are GitHub's first-party actions maintained by the Actions team. No third-party action invocations. Version pinned to major (`@v4`); automatic security patches via GitHub Dependabot if enabled at repo level. |
| T-04-03 | Information Disclosure | GITHUB_TOKEN leaking to job logs | mitigate | No `secrets.` references in ci.yml. Default permissions are read-only for public repos. No `echo $GITHUB_TOKEN` or debug statements. |
| T-04-04 | Denial of Service | Matrix job stacking exhausting CI minutes | mitigate | `concurrency.group: ci-${{ github.ref }}` + `cancel-in-progress: true` cancels older runs on the same ref. Force-pushes do not pile up. |
| T-04-05 | Tampering | Workflow triggered by untrusted PR from a fork running on main | accept | Public-repo convention: PR workflows run under the PR's code but with the base repo's default token (read-only). A fork PR cannot access secrets and cannot write to the base repo. GitHub's security model handles this; no custom protection needed in Phase 1. Phase 5 (if publishing) will need explicit `pull_request_target` review. |
| T-04-06 | Tampering | Running `npm test` executes attacker-controlled code from a PR | accept | This is the standard PR CI model — every repo that runs tests on PRs has this trust boundary. Mitigation is code review before merge, not a workflow change. Phase 1 has no secrets to exfiltrate (no tokens, no env vars), so the blast radius is limited to CI minutes. |
</threat_model>
