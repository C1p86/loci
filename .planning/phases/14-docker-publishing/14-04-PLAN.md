    ```markdown
    # xci Release Runbook

    This runbook is for release operators — the human who decides to cut a release. Follow it **in order**. Assume a fresh clone of main.

    ## Prerequisites (one-time setup per operator)

    - [ ] Write access to the `loci` repo
    - [ ] You are able to merge PRs on main
    - [ ] `NPM_TOKEN` is set as a repo secret (Settings → Secrets and variables → Actions) — pending todo from Phase 7 STATE.md
    - [ ] Repo Settings → Actions → General → "Allow GitHub Actions to create and approve pull requests" is enabled — pending todo
    - [ ] Branch protection on main: `build-test-lint` (6 matrix jobs), `fence-gates`, `integration-tests`, `web-e2e` are Required Status Checks — pending todo

    ## First release (v2.0.0) — one-time dry-run

    Per CONTEXT D-22, the first release MUST be a release-candidate run end-to-end before the real v2.0.0 tag.

    1. [ ] On a branch `rc/v2.0.0-rc.1`, run `pnpm changeset` and describe the milestone in one line (e.g. "Remote CI milestone: server + web + Docker")
    2. [ ] Open PR, merge to main (CI green required)
    3. [ ] Changesets action opens a "Version Packages" PR
    4. [ ] Review the generated version bump — should be `2.0.0-rc.1` for xci, @xci/server, @xci/web (fixed versioning)
    5. [ ] Merge the Version Packages PR
    6. [ ] Changesets action runs the real publish step + creates git tag `v2.0.0-rc.1`
    7. [ ] The new tag triggers `.github/workflows/docker.yml`:
         - Watch the workflow run
         - Verify smoke step passes (log line `[smoke:PASS] all 13 steps green`)
         - Verify Trivy step passes (no HIGH/CRITICAL)
         - Verify image pushed as `ghcr.io/<owner>/xci-server:v2.0.0-rc.1` (only versioned tag for pre-release)
    8. [ ] Verify npm pre-release publication:
         - `npm view xci version` → should show `2.0.0-rc.1` (if `--tag next` was used in changesets config; check changeset pre/exit flow)
         - `npm view @xci/server version` → `2.0.0-rc.1`
         - `npm view @xci/web version` → `2.0.0-rc.1`
    9. [ ] Test install on a throwaway machine:
         - `npm i -g xci@next` (or `@2.0.0-rc.1`) → `xci --version` prints 2.0.0-rc.1
         - `docker pull ghcr.io/<owner>/xci-server:v2.0.0-rc.1 && docker run --rm <img> id -u` → 10001
    10. [ ] If any step failed, fix and tag v2.0.0-rc.2, repeat
    11. [ ] Once rc.N is clean, run the "Subsequent release" flow for v2.0.0

    ## Subsequent release

    1. [ ] Ensure main is green (CI badge passing)
    2. [ ] `pnpm changeset status` locally to review pending changesets
    3. [ ] Merge Changesets "Version Packages" PR (opened automatically on main push)
    4. [ ] Watch `.github/workflows/release.yml` — pre-publish validation (typecheck/lint/build/test/dry-run) + Changesets action runs publish
    5. [ ] Changesets creates tag `vX.Y.Z`
    6. [ ] Tag push triggers `.github/workflows/docker.yml`:
         - Build + smoke + Trivy + push to ghcr.io with tags `latest`, `vX.Y.Z`, `vX.Y`, `vX`
    7. [ ] Post-release smoke (5-min manual):
         - [ ] `npm i -g xci@latest && xci --version` → new version
         - [ ] `docker pull ghcr.io/<owner>/xci-server:latest` succeeds
         - [ ] `docker run --rm ghcr.io/<owner>/xci-server:latest node -e "console.log('ok')"` prints `ok`
    8. [ ] Announce the release (manual; out of runbook scope)

    ## If a release breaks

    - npm: `npm deprecate <pkg>@<version> "broken; use <X.Y.Z+1>"` (cannot unpublish after 72h)
    - ghcr.io: delete the bad image tag from ghcr.io UI; consumers pinned to a bad tag should re-pin
    - Cut a patch release immediately — do not leave `latest` pointing at broken
    - Prefer forward-fix over rollback (Changesets is forward-only)

    ## Security checklist (each release)

    - [ ] NPM_TOKEN has not leaked (grep last 30 days of CI logs — GitHub masks secrets by default)
    - [ ] No new HIGH/CRITICAL Trivy findings were suppressed (review `ignore-unfixed` bypasses)
    - [ ] MEK rotation cadence: document when production MEK was last rotated (rotate annually or after any secret exposure)

    ## Links

    - Dev stack: [`docker-compose.yml`](../docker-compose.yml)
    - Dockerfile: [`packages/server/Dockerfile`](../packages/server/Dockerfile)
    - Smoke script: [`scripts/smoke.mjs`](../scripts/smoke.mjs)
    - Phase 14 context: [`.planning/phases/14-docker-publishing/14-CONTEXT.md`](../.planning/phases/14-docker-publishing/14-CONTEXT.md)
    ```
