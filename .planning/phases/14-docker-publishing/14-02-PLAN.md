    **Step 1 — Create `docker-compose.yml` at repo root** (D-08):
    ```yaml
    # Local dev stack. NOT suitable for production.
    # Prod deployments should use orchestrator of choice (k8s/ECS/etc.) with
    # the published ghcr.io/xci/server image + external managed Postgres + real SMTP.

    services:
      postgres:
        image: postgres:16
        restart: unless-stopped
        environment:
          POSTGRES_USER: xci
          POSTGRES_PASSWORD: xci_dev_password
          POSTGRES_DB: xci
        ports:
          - "5432:5432"
        volumes:
          - pgdata:/var/lib/postgresql/data
        healthcheck:
          test: ["CMD", "pg_isready", "-U", "xci", "-d", "xci"]
          interval: 5s
          timeout: 5s
          retries: 10
          start_period: 10s

      mailhog:
        image: mailhog/mailhog:v1.0.1
        restart: unless-stopped
        ports:
          - "1025:1025"   # SMTP ingress
          - "8025:8025"   # Web UI — open http://localhost:8025 to inspect emails

      server:
        build:
          context: .
          dockerfile: packages/server/Dockerfile
        restart: unless-stopped
        depends_on:
          postgres:
            condition: service_healthy
          mailhog:
            condition: service_started
        env_file:
          - .env
        environment:
          # These override anything in .env — they are fixed by the compose network topology
          DATABASE_URL: postgres://xci:xci_dev_password@postgres:5432/xci
          SMTP_HOST: mailhog
          SMTP_PORT: "1025"
          EMAIL_TRANSPORT: smtp
          SMTP_FROM: xci-dev@localhost
          # NODE_ENV=production is baked into image; override to development if you want pino-pretty logs
          NODE_ENV: production
        ports:
          - "3000:3000"
        healthcheck:
          test: ["CMD", "curl", "-fsS", "http://localhost:3000/api/healthz"]
          interval: 15s
          timeout: 5s
          retries: 3
          start_period: 30s

    volumes:
      pgdata:
    ```

    **Why `env_file` + explicit `environment`**: env_file provides XCI_MASTER_KEY / SESSION_COOKIE_SECRET / PLATFORM_ADMIN_EMAIL (dev values from .env.example copied to .env); the `environment:` block fixes network-topology vars (DATABASE_URL points at the compose-internal `postgres` hostname, not localhost).

    **Why `mailhog/mailhog:v1.0.1` (pinned)**: mailhog has no 2.x; v1.0.1 is the last tagged stable.

    # xci dev environment — copy to .env for `docker compose up`.

    # ============================================================================

    # --- Session cookie secret (REQUIRED, min 32 chars) ---
    # DEV ONLY. Generate a fresh one for production:
    #   node -e "console.log(require('node:crypto').randomBytes(48).toString('hex'))"
    SESSION_COOKIE_SECRET=dev-session-secret-must-be-at-least-32-chars-long

    # --- Master encryption key (REQUIRED, base64 of 32 random bytes = 44 chars ending in '=') ---
    # DEV ONLY. REGENERATE FOR PRODUCTION:
    #   node -e "console.log(require('node:crypto').randomBytes(32).toString('base64'))"
    # Losing this key in production makes ALL encrypted org secrets unrecoverable.
    # Rotating this key in production requires calling POST /api/admin/rotate-mek (see runbook).
    XCI_MASTER_KEY=AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA

    # --- Platform admin email (REQUIRED) ---
    # The ONE account allowed to call POST /api/admin/rotate-mek.
    # Must match a real user email that has signed up. Case-insensitive.
    PLATFORM_ADMIN_EMAIL=admin@localhost

    # --- Database URL (REQUIRED in prod; dev compose overrides this) ---
    # Format: postgres://user:password@host:port/database
    # In docker compose, this is overridden to postgres://xci:xci_dev_password@postgres:5432/xci
    DATABASE_URL=postgres://xci:xci_dev_password@localhost:5432/xci

    # --- Email transport (REQUIRED) ---
    # Options: log | stub | smtp
    # - log:   logs email bodies to stdout (best for local dev without mailhog)
    # - stub:  discards emails (CI/test usage)
    # - smtp:  uses SMTP_HOST+SMTP_PORT+SMTP_FROM (+ optional SMTP_USER/SMTP_PASS)
    # Compose overrides to smtp pointing at the mailhog service.
    EMAIL_TRANSPORT=smtp

    # --- SMTP config (REQUIRED only if EMAIL_TRANSPORT=smtp) ---
    # Compose overrides SMTP_HOST, SMTP_PORT, SMTP_FROM to point at mailhog container.
    SMTP_HOST=localhost
    SMTP_PORT=1025
    SMTP_USER=
    SMTP_PASS=
    SMTP_FROM=xci-dev@localhost

    # --- Optional ---
    # NODE_ENV=production       # image default; set to development to enable pino-pretty logs
    # PORT=3000                 # image default
    # LOG_LEVEL=info            # fatal|error|warn|info|debug|trace
    # LOG_RETENTION_INTERVAL_MS=86400000   # 24h retention cleanup cadence
    # WEB_STATIC_ROOT=/app/web  # image default — serve @xci/web SPA at /
    ```

    ## Running in Docker

    `@xci/server` ships as a production Docker image at `ghcr.io/xci/server:<tag>` (see [repo root README](../../README.md#docker-quick-start) for the release tag scheme).

    ### Dev stack (docker compose)

    From the repo root:

    ```bash
    cp .env.example .env             # copy dev defaults; never commit .env
    docker compose up -d --build     # builds Dockerfile, starts postgres + mailhog + server
    docker compose logs -f server    # follow server logs
    ```

    The stack is up when `docker compose ps` reports `server` status `healthy`.

    | Endpoint | URL |
    |----------|-----|
    | Health | http://localhost:3000/api/healthz |
    | Web SPA | http://localhost:3000/ |
    | API | http://localhost:3000/api/* |
    | WS (agent) | ws://localhost:3000/ws/agent |
    | WS (log subscribe) | ws://localhost:3000/ws/orgs/:orgId/runs/:runId/logs |
    | Webhooks | http://localhost:3000/hooks/:plugin/:orgToken |
    | Badge | http://localhost:3000/badge/:orgSlug/:taskSlug.svg |
    | MailHog UI | http://localhost:8025 |

    To clean up (including the Postgres data volume):

    ```bash
    docker compose down -v
    ```

    ### What's inside the image

    - Base: `node:22-slim` (glibc; required for `@node-rs/argon2` prebuilt binaries)
    - User: `xci` (UID `10001:10001`, non-root)
    - Healthcheck: `curl -fsS http://localhost:3000/api/healthz` every 15s
    - Migrations: applied at boot via the programmatic Drizzle migrator (no `drizzle-kit` in the runtime image)
    - Web SPA: `@xci/web` prebuilt bundle served by `@fastify/static` from `/app/web` (disable by setting `WEB_STATIC_ROOT=''`)

    ### Production deployment notes

    - Override ALL secrets in `.env.example` before running in production. In particular:
      - Regenerate `XCI_MASTER_KEY` (`node -e "console.log(require('node:crypto').randomBytes(32).toString('base64'))"`)
      - Regenerate `SESSION_COOKIE_SECRET`
      - Point `DATABASE_URL` at a managed Postgres (not the compose container)
      - Configure real SMTP (`EMAIL_TRANSPORT=smtp` + `SMTP_HOST/PORT/USER/PASS/FROM`)
      - Set `PLATFORM_ADMIN_EMAIL` to a real user email
    - Terminate TLS at your reverse proxy (nginx / Caddy / cloud LB). The image does not handle HTTPS directly.
    - Pin to a specific version tag (`ghcr.io/xci/server:v2.0.0`). Do not rely on `:latest` in production.
    - On SIGTERM the server drains WS connections, flushes the log batcher, and exits 0 (PKG-06). Orchestrators should give it at least 30s to drain.

    ### Rebuilding from source

    ```bash
    # From repo root
    docker build -f packages/server/Dockerfile -t xci/server:dev .
    ```

    This triggers the two-stage build: `node:22` builder runs `pnpm install --frozen-lockfile` + `pnpm turbo run build --filter=@xci/server --filter=@xci/web` + `pnpm deploy --prod`; `node:22-slim` runtime COPYs the pruned `node_modules` + `dist/` + `drizzle/` + web bundle.
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| .env file → committed artifacts | Developers must copy `.env.example` to `.env`; `.env` is gitignored. Safety relies on correct gitignore + operator discipline. |
| Dev compose → production | docker-compose.yml is explicitly dev-only; README says "NOT suitable for production". Risk: operator runs dev compose file in prod. |
| Local Postgres → attackers | Postgres port 5432 published to host; credentials are placeholders but known. Only a risk if developer machine is on an untrusted network. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-02-01 | Information Disclosure | .env file committed by mistake | mitigate | `.gitignore` already lists `.env` patterns; .env.example is the committed doc. Add lint/precommit check? DEFERRED — rely on gitignore + code review. |
| T-14-02-02 | Spoofing | Dev secrets shipped in .env.example could be reused in prod | mitigate | Every secret line in .env.example has an explicit warning comment ("DEV ONLY. REGENERATE FOR PRODUCTION:"). Runbook (Plan 14-04) reinforces. |
| T-14-02-03 | Elevation of Privilege | Postgres admin credentials in docker-compose.yml plaintext | accept | Dev-only file; credentials are well-known placeholders (`xci_dev_password`). No real-user impact. |
| T-14-02-04 | Denial of Service | `docker compose up` in a loop stacking volumes | accept | pgdata volume name fixed; compose idempotent. Not an attack vector. |
</threat_model>
