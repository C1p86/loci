<objective>
Expose Secrets CRUD (SEC-04) + Secret Audit Log listing (SEC-07) with the ironclad architectural invariant that NO endpoint ever returns a plaintext secret value. Every mutation transparently performs AES-256-GCM envelope encryption (via Plan 09-02 crypto) and records an audit entry in the SAME database transaction (D-22).

Purpose:
- SEC-04: Owner/Member can create/update/delete secrets; any logged-in org member can list metadata; Viewer cannot mutate.
- SEC-01..03: encryption happens inside the repo (Plan 09-03) — route handlers never touch ciphertext, dek, or mek; they pass `value` straight to `repos.forOrg(orgId).secrets.create({name, value, ...})`.
- SEC-07: paginated audit log visible to Owners only; Phase 13 will surface this in the UI; this plan builds the JSON endpoint.
- The "no plaintext in response" invariant is CI-enforced via an integration test that parses every response body and asserts no leak of `value`/`ciphertext`/`iv`/`auth_tag`/`dek`/`mek` keys.

Output:
- 6 route handler files + 1 barrel + 6 integration tests (5 per-endpoint + 1 invariant guard)
- Routes mounted at /api/orgs/:orgId/secrets and /api/orgs/:orgId/secret-audit-log under the existing /api/orgs prefix
- CSRF + requireAuth + role guards applied uniformly; Owner-only for DELETE and audit-log GET; Owner/Member for create/update; any-member for metadata list
- Zero plaintext leak confirmed by dedicated invariant test
</objective>
