# Phase quick-260421-g99: Add optional cwd field to xci aliases Summary

**Optional `cwd` on every alias kind with parent→child inheritance, lenient placeholder resolution, absolute-path rewrite at the CLI boundary, per-step/per-entry spawn-site support, and run-header / dry-run / verbose display with secret-value redaction.**

- **Completed:** 2026-04-21T12:12:00Z (approx.)

- **Files modified:** 14 (2 created, 12 modified)
- **Test count:** 453 → 501 (+48 tests total across 4 test files)

## Accomplishments

### Task 1 — DSL + resolver + params tracking (commit `aef961c`)

- `types.ts`: added `readonly cwd?: string` to every `CommandDef` variant (single, sequential, parallel, for_each, ini); to the `cmd` and `ini` variants of `SequentialStep`; to `ExecutionPlan` single/ini; to each entry of `parallel.group`. `set` variant of `SequentialStep` deliberately has no cwd (it does not spawn).
- `commands/normalize.ts`: new `parseCwd(aliasName, obj)` helper — rejects non-string cwd with exact message `"cwd must be a string"`. Invoked in all 5 branches of `normalizeObject` (ini, for_each, steps/sequential, parallel, single/platform-fallthrough). String-form and array-form shorthand continue to have no cwd.
- `resolver/index.ts`: new `computeEffectiveCwd(def, config, parentCwd)` — returns lenient-interpolated own cwd, else parent. Threaded `parentCwd?: string` as 6th optional param in both `resolveAlias` and `resolveToStepsLenient`. Each kind branch emits `effectiveCwd` into the plan: single → plan.cwd, sequential → inline step cwd + recursive child inheritance, parallel → entry cwd from subPlan.cwd or inline effectiveCwd, for_each → expanded step/entry cwd, ini → plan.cwd. Top-level `resolver.resolve` call passes `undefined`.
- `resolver/params.ts`: added one shared `trackUsage(extractPlaceholders(def.cwd))` block before the `switch(def.kind)` so `${...}` in cwd surfaces as a required param via `MissingParamsError` on ALL kinds uniformly.
- 37 new tests across `commands.test.ts` (load-time validation, 5 kinds × non-string rejection), `resolver.test.ts` (14 cases: interpolation, inheritance, child-wins, inline/alias mixing, for_each parallel vs steps, parallel entry cwd, ini plan cwd, grandparent-through-middle), and `params-nested-placeholders.test.ts` (5 kinds × missing placeholder surfacing).

### Task 2 — Executor wiring + output display (commit `0aada91`)

- New `packages/xci/src/executor/cwd.ts` exporting `resolveAbsoluteCwds(plan, projectRoot)` — switch over plan.kind; relative cwds get `resolvePath(projectRoot, cwd)`, absolutes pass through unchanged, undefined stays undefined, `set` steps untouched.
- `cli.ts`: imports `resolveAbsoluteCwds`; after `resolver.resolve` calls it with projectRoot; threads the rewritten plan through `printRunHeader` (new `projectRoot` last positional arg), `printVerboseCommand`, `printDryRun`, `appendExtraArgs`, `executor.run`, `runWithDashboard`.
- `executor/index.ts`: single case uses `plan.cwd ?? cwd`; ini case uses `plan.cwd ?? cwd` both for spawn-context AND for resolving relative `plan.file` (via `isAbsolute`/`resolvePath` imported at top). Re-exports `resolveAbsoluteCwds` for cli.
- `executor/sequential.ts`: `stepSpawnCwd = step.cwd ?? cwd` used for both `runAndCapture` (capture path) and `runSingle` (non-capture path); ini step branch resolves relative `step.file` against `step.cwd ?? cwd`.
- `executor/parallel.ts`: destructures `cwd: entryCwd` from group entry in the `map` body, uses `entryCwd ?? cwd` as execa's `cwd`.
- `executor/output.ts`: new `redactCwd(cwd, secretValues)` helper + `topLevelCwd(plan)` selector (single/ini → plan.cwd, sequential → first spawn step's cwd, parallel → first entry's cwd). `printRunHeader` emits a dim `cwd: <path>` line between variables and steps (hidden when cwd matches projectRoot); per-step/per-entry divergence appears as inline dim ` (cwd: <path>)` suffix. `printDryRun` and `printVerboseCommand` mirror the pattern; verbose is exhaustive (shows per-step/entry when divergent).
- `tui/dashboard.ts`: mirrored the spawn-site pattern in `execSingle` (plan.cwd ?? cwd), `execSequential` cmd steps (step.cwd ?? cwd), `execSequential` ini step (resolves file path against step.cwd ?? cwd), and `execParallel` (entry.cwd ?? cwd).
- 11 new unit + integration tests in `executor/__tests__/cwd.test.ts`: pure `resolveAbsoluteCwds` transform over all 4 plan kinds + real-tempdir integration tests that spawn `node -e process.cwd()` and assert the child landed in the declared cwd (single, sequential with different per-step cwds, parallel with different per-entry cwds).
