<objective>
Re-interpolate the `cwd` field against the right variables at the right time so that a step's working directory can be driven by a value that only exists mid-sequence (captured) OR by a `for_each` loop variable (per-iteration). Three coupled fixes ship together (none alone is sufficient):

1. **Resolve-time corruption (cwd.ts `toAbs`):** A cwd still containing `${...}` is not absolute, so `toAbs` joins it to projectRoot → `projectRoot\${P4_WORKSPACE_ROOT}`, corrupting it before the variable is even known. Fix: DEFER absolutization when an unresolved `${` placeholder remains — leave the string intact so it survives to runtime.

2. **No runtime re-interpolation (sequential.ts):** Every step kind uses `step.cwd ?? cwd` directly and never re-interpolates cwd against `mergedValues` (the way it already does for argv, ini/uproject file, unreadonly path, and xci alias/project/args). Fix: introduce a single shared helper `resolveRuntimeCwd(rawCwd, values, baseCwd)` in cwd.ts (interpolate → absolutize against base) and use it at all ~5 call sites. This covers `set` / `capture` / `prompt` captured vars.

3. **for_each loop var never reaches cwd (resolver/index.ts `bakeLoopVarIntoRawArgv`):** Inside a `for_each` body, the loop var is baked into each step's `rawArgv` at resolve time but NOT into `step.cwd`, and it is never added to `capturedVars` at runtime. So a body step with `cwd: "${LOOP_VAR}"` would hit `UndefinedPlaceholderError` (fix #2 cannot help — the loop var is not a captured var). Fix: bake the loop var into `step.cwd` too, at resolve time, for ANY body step that carries a cwd (not just command-variant steps with rawArgv).

**Combined ordering (must be preserved so deferral is not broken):**
`bakeLoopVarIntoRawArgv` (resolve time, loop var → rawArgv + cwd) → `resolveAbsoluteCwds` (absolutize, but DEFER if a `${` still remains) → runtime `resolveRuntimeCwd` (captured vars) → absolutize against base. After baking, a cwd that was purely `${LOOP_VAR}` no longer contains `${`, so it absolutizes normally; a cwd that still references a captured var keeps its `${` and is deferred to runtime.

Purpose: Captured/runtime variables AND for_each loop variables can each drive a step's working directory, by their respective mechanisms.
Output: Updated cwd.ts + sequential.ts + resolver/index.ts, plus regression tests; rebuilt and reinstalled xci.
</objective>
