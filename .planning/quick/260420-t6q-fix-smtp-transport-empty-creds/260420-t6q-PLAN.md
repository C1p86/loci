<objective>
Fix the SMTP transport factory so empty-string `SMTP_USER` / `SMTP_PASS` values are treated identically to `undefined` — i.e. nodemailer is configured WITHOUT `auth`, so it does not attempt `AUTH PLAIN` against mailhog (or any unauthenticated relay).

Purpose: Unblocks signup / password-reset email delivery in the dev docker-compose stack. Root cause already diagnosed (see planning_context): `.env` ships with `SMTP_USER=` as an empty string, the current `!== undefined` guard passes, and nodemailer throws `Missing credentials for "PLAIN"`.

Output: Patched `transport.ts` + one new test case in `transport.test.ts` asserting that empty credentials produce an auth-less transporter.
</objective>

**Edit 1 — `packages/server/src/email/transport.ts` (around lines 51-58):**

Change the SMTP user guard from an `!== undefined` check to a truthy check so empty strings short-circuit out of the auth block. Replace:

```typescript
...(cfg.SMTP_USER !== undefined && {
  auth: { user: cfg.SMTP_USER, pass: cfg.SMTP_PASS ?? '' },
}),
```

with:

```typescript
...(cfg.SMTP_USER && {
  auth: { user: cfg.SMTP_USER, pass: cfg.SMTP_PASS ?? '' },
}),
```

Rationale (do NOT add as a source comment — CLAUDE.md §project skills "no unnecessary comments"):
- Empty string is now treated identically to `undefined` — no `auth` key is spread into the nodemailer options.
- `cfg.SMTP_PASS ?? ''` is preserved for the case where `SMTP_USER` is set but `SMTP_PASS` is not (e.g. token-based SMTP where only the user field matters).
- Type of the spread expression changes from `false | { auth: ... }` to `'' | undefined | { auth: ... }`, which TypeScript still accepts for an object spread (falsy values spread to nothing).

No other lines change. Do NOT touch the `if (!cfg.SMTP_HOST || !cfg.SMTP_FROM)` guard above, do NOT touch the `'log'` or `'stub'` branches, do NOT change `TransportConfig`, do NOT change `.env.example`, do NOT change any route or the env schema.
