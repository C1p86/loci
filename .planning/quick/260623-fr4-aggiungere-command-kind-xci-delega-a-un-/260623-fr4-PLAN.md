<objective>
Add a new DSL command kind `xci` to the xci CLI (packages/xci) that delegates execution to ANOTHER xci instance (typically a different, nested project). It spawns the xci binary itself with `stdio: 'inherit'` and `cwd` = the target project root, forwarding an `alias` and pass-through `args`, and sets an `XCI_NESTING_DEPTH` env var so the inner xci attenuates its own output (no real-time tail cursor-moves, no terminal-title OSC sequences, no desktop notifications).

This fixes two real bugs the user hit when an xci script invoked another xci as a plain `cmd` leaf:
1. **Garbled logs** — the inner xci's banners/colored prefixes/terminal-title OSC/real-time tail (`\x1b[A\x1b[2K`) collide with the outer xci re-piping and re-tailing that output.
2. **Parent hangs after child finishes** — the outer xci always creates a log file, so runSingle pipes the child's streams and `await proc` only resolves once the streams EOF. A background grandchild holding the pipe write-end means the streams never close and the outer xci hangs forever. `stdio: 'inherit'` makes the parent wait only on process exit, so this is structurally impossible.

Purpose: nested xci becomes a first-class kind instead of an ordinary leaf command, eliminating both collisions.
Output: a fully wired `xci` kind — types, normalize (via `kind: xci` discriminator), resolver, a new `executor/xci-delegate.ts` (inherit-based spawn) + `executor/nesting.ts` (depth chokepoint), executor dispatch, sequential-step support, output attenuation in single.ts/output.ts, help/list/dry-run/verbose output, TUI label, unit + e2e tests, README docs, and a changeset.
</objective>

### YAML schema (detected by `kind: xci`, NOT by a top-level key like uproject)
```yaml
deploy-other:
  kind: xci
  project: C:/path/to/other-project   # optional — target project root; default = current cwd
  alias: deploy                        # required — alias to run in the target project
  args: [env=prod]                     # optional — KEY=VALUE / pass-through args forwarded to the inner xci
```
Also supported as a sequential step (inside a `kind: sequential` steps list), mirroring how uproject works as
both a top-level kind AND an inline sequential step.

### CommandDef / ExecutionPlan / SequentialStep shapes (types.ts)
Add this member to ALL THREE unions (CommandDef, ExecutionPlan, SequentialStep). The SequentialStep variant
also carries `breadcrumb?`. The ExecutionPlan + SequentialStep `args` are post-interpolation; the CommandDef
`args` is raw.
```
{
  readonly kind: 'xci';
  readonly project?: string;          // target project root (relative/absolute/${...}); default current cwd
  readonly alias: string;             // alias to run in the target project (required)
  readonly args?: readonly string[];  // forwarded argv (KEY=VALUE overrides + pass-through)
  readonly description?: string;      // CommandDef only
  readonly params?: Readonly<Record<string, ParamDef>>; // CommandDef only
  readonly cwd?: string;              // standard cwd field (all kinds)
  readonly breadcrumb?: readonly string[]; // SequentialStep only
}
```
NOTE: `cwd` and `project` are distinct. `cwd` is the standard alias-level cwd (kept for uniformity / cwd
inheritance). `project` is the delegate target. **The delegate spawn uses `project` (made absolute) as its
spawn cwd; if `project` is absent, fall back to `plan.cwd ?? options.cwd` (the effective project root).**

### Re-invoking the xci binary (cross-platform — LOCKED approach)
Spawn `process.execPath` (the Node binary) with `[process.argv[1], aliasToRun, ...args]` where
`process.argv[1]` is the currently-running entry script (the resolved `dist/cli.mjs`, both for global installs
and from-source/dev). This avoids PATH/`.cmd`-shim resolution problems on Windows and works identically when
installed globally and when run from source. Use execa with `stdio: 'inherit'`. Do NOT shell out to a bare
`xci` command (Windows `.cmd` shim + PATH ambiguity).
  - argv: `[process.execPath, process.argv[1], alias, ...args]`
  - cwd: resolved target project root (project abs > plan.cwd > options.cwd)
  - env: `{ ...process.env, ...env, XCI_NESTING_DEPTH: String(getNestingDepth() + 1) }`
  - To make this testable, `runXciDelegate` accepts the entry script path as a parameter (default
    `process.argv[1]`) so the unit test can assert argv without spawning.
